# 通知系统 API 文档与前端类型映射约定

## 1. 文档目的

本文档用于约定插件通知系统与外部通知平台之间的接口行为，以及插件前端对通知类型的统一展示规则。

适用范围：

- 外部通知推送平台
- `astrbot_plugin_proactive_chat`
- `astrbot_plugin_disaster_warning`
- 插件 Web 管理端前端通知中心

本文档基于当前已确认的远端 JSON 返回体编写。

---

## 2. 接口概览

### 2.1 请求方式

- 方法：`GET`
- 鉴权：当前无需鉴权
- 响应格式：`application/json`
- 响应主体：JSON 数组

### 2.2 当前接口路径与请求参数

```text
/api/v1/{app_slug}/notifications/updates?plugin_version={plugin_version}
```

其中：

- `plugin_version`：必填，插件当前版本号（由插件侧自动附加）
- `{app_slug}`：外部通知平台中为某个应用分配的唯一标识
- 对于本插件，应由插件配置或固定常量提供

### 2.3 当前已确认示例

```json
[
{
"id": 2,
"app_id": 2,
"title": "主动消息测试",
"content": "# 🌿 守望内心的安宁\\n\\n> 在这个信息纷扰的世界里，能够守护住内心的片刻宁静，本身就是一种**了不起的成就**。\\n\\n### 生活的节奏\\n我们不需要时刻保持**冲刺**的状态，有时候，慢下来去观察：\\n* 一朵花的**开合**\\n* 雨滴落在窗棂的**声音**\\n\\n反而能找回那份被遗忘的**生命力**。真正的力量往往不在于喧嚣，而在于那份看清前路后的**笃定与从容**。\\n\\n---\\n*生成于 2026年3月*",
"content_format": "markdown",

}

```

## 3. 响应数据结构定义

响应主体为数组，每个元素表示一条通知。

### 3.1 通知对象字段说明

| 字段名 | 类型 | 必填 | 说明 |
| :------- | :--------- | :--- | :---------------------------- |
| `id` | `number` | 是 | 通知唯一 ID，用于去重、排序、已读状态映射 |
| `app_id` | `number` | 是 | 外部通知平台中的应用 ID |
| `title` | `string` | 是 | 通知标题 |
| `content` | `string` | 是 | 通知正文 |
| `content_format` | `string` | 否 | 正文格式，支持 `text` / `markdown`；缺省按 `text` 处理 |
| `type` | `string` | 是 | 通知类型代码，前端将基于此进行类型映射与样式展示 |
| `created_at` | `string` | 是 | 通知创建时间，要求为 ISO 8601 时间字符串 |
| `updated_at` | `string` | 否 | 通知更新时间，若提供应为 ISO 8601 时间字符串 |
| `is_active` | `boolean` | 是 | 通知是否为生效状态；插件侧仅展示生效通知 |

### 3.2 字段约束说明

#### `id`

- 必须在同一应用下保持唯一
- 一经创建不应复用
- 插件侧将基于 `id` 维护本地已读状态

#### `title`

- 建议长度：1 ~ 80 个字符
- 应能独立概括通知主题
- 不建议为空字符串

#### `content`

- `content_format = text` 时按纯文本展示
- `content_format = markdown` 时按 Markdown 渲染
- 为了兼容历史数据，前端会对明显的 Markdown 块级语法（标题/引用/列表/分隔线/代码围栏/表格）做自动识别渲染
- 链接会被加上安全属性（`target="_blank"`、`rel="noopener noreferrer"`），并限制危险协议
- 当前前端支持的重点 Markdown 能力包括：标题、段落、引用、无序列表、有序列表、分隔线、行内代码、围栏代码块、表格、强调/删除线

#### `content_format`

- 可选值：`text`、`markdown`
- 缺省、空值或未知值应按 `text` 处理
- 插件后端会将 `plain` / `plaintext` 归一化为 `text`

#### `type`

- 建议使用全大写英文代码
- 插件前端不会直接展示原始代码，而会按映射表转换为可读标签

#### `created_at`

- 必须为可被 JavaScript `Date` 正确解析的 ISO 8601 字符串
- 推荐使用 UTC 时间，例如：`2026-03-27T09:00:20.218Z`

#### `updated_at`

- 必须为可被 JavaScript `Date` 正确解析的 ISO 8601 字符串
- 推荐使用 UTC 时间，例如：`2026-03-27T09:00:20.218Z`

#### `is_active`

- `true`：通知处于有效状态，可在插件通知中心展示
- `false`：通知已撤回、停用或不再展示，插件同步后应从展示列表中移除

## 4. 插件侧处理约定

### 4.1 过滤规则

插件后端在同步远端通知后，应执行以下处理：

1. 仅保留 `is_active === true` 的通知
2. 丢弃缺少关键字段的非法项
3. 按 `created_at` 倒序排序
4. 使用 `id` 作为本地通知唯一键

### 4.2 非法数据判定建议

若通知对象存在以下情况之一，插件后端应视为非法数据并忽略该项：

- 缺少 `id`
- 缺少 `title`
- 缺少 `content`
- 缺少 `type`
- 缺少 `created_at`
- `created_at` 无法被解析为有效时间
- `is_active` 不是布尔值

### 4.3 本地已读状态约定

插件本地维护通知已读状态，远端接口当前不承担已读同步职责。

本地已读状态建议按如下方式维护：

- 键：通知 `id`
- 值：是否已读

示例：

```json
{
  "read_map": {
    "2": true,
    "5": false
  }
}
```

### 4.4 展示顺序约定

通知中心默认按 `created_at` 倒序展示：

- 最新通知显示在最前面
- 相同时间可按 `id` 倒序作为次级排序依据

## 5. 前端通知类型映射约定

为了保证团队在不同页面、不同版本中的通知类型表现一致，前端应统一遵循以下类型映射规则。

### 5.1 标准类型代码

建议远端通知平台优先使用以下标准类型：

| 类型代码 | 中文标签 | 语义说明 | 建议视觉等级 |
| :-- | :-- | :-- | :-- |
| `UPDATE` | 更新通知 | 新版本发布、功能上线、版本升级说明 | `info` |
| `BUGFIX` | 修复通知 | Bug 修复、热修复、兼容性修正 | `success` |
| `NOTICE` | 公告通知 | 普通公告、迁移说明、提醒信息、运营通知 | `warning` |
| `SECURITY` | 安全通知 | 安全风险、漏洞修复、紧急升级提示 | `error` |
| `TEST` | 测试通知 | 内部联调、连通性测试、灰度验证 | `default` |

### 5.2 兼容别名约定

当前阶段仅保留最小必要兼容：

| 原始类型 | 归一化到 | 说明 |
| :-- | :-- | :-- |
| `INFO` | `NOTICE` | 普通信息类通知，前端统一显示为“公告通知” |

若未命中标准类型和兼容别名，则进入兜底规则。

### 5.3 未识别类型兜底规则

若前端遇到未在标准列表中的 `type`：

- 标签统一显示为：`其他通知`
- 视觉等级统一按：`default`
- 原始 `type` 可在调试信息或开发日志中保留，但不直接暴露给普通用户

### 5.4 显示层统一约定

前端展示通知卡片时，每条通知至少应显示：

- 标题
- 类型标签（由 `type` 映射而来）
- 创建时间
- 正文
- 已读/未读状态

### 5.5 建议的前端映射表示例

```js
const NOTIFICATION_TYPE_META = {
    UPDATE: { label: '更新通知', tone: 'info' },
    BUGFIX: { label: '修复通知', tone: 'success' },
    NOTICE: { label: '公告通知', tone: 'warning' },
    SECURITY: { label: '安全通知', tone: 'error' },
    TEST: { label: '测试通知', tone: 'default' },
};

const NOTIFICATION_TYPE_ALIASES = {
    INFO: 'NOTICE',
};
```

当 `type` 不在标准表和别名表内时：

```js
{ label: '其他通知', tone: 'default' }
```

## 6. 当前版本展示行为约定

当前版本中，插件前端通知中心应遵循以下行为：

1. 仅展示激活通知
2. 支持 `content_format=text` 纯文本展示
3. 支持 `content_format=markdown` Markdown 渲染
4. 对明显 Markdown 块级语法支持自动识别渲染（兼容历史数据）
5. Markdown 展示已接入管理端主题配色，标题、引用、链接、行内代码、代码块、表格会使用统一的前端视觉语言
6. 当第三方 Markdown 解析器不可用时，前端仍会使用内置 fallback 渲染器保留主要结构语义
7. 不支持服务端已读同步
8. 不支持分页
9. 不支持按类型筛选
10. 支持本地标记单条已读
11. 支持本地全部标记为已读
12. 支持显示未读数量
13. 支持通过插件后端主动刷新远端通知
14. 支持通过插件 WebSocket 广播实时更新

### 6.1 Markdown 展示主题约定

为保证通知中心中的 Markdown 内容与管理端原生界面保持统一，前端展示层应遵循以下规则：

- 标题层级使用管理端现有主文本色与主色强调，不引入第三套内容主题
- 行内代码应映射为轻量 pill 风格，优先使用 `surface / primary` 相关色系
- 围栏代码块应映射为卡片内次级 panel 风格，保留横向滚动能力
- 表格应包裹在可横向滚动容器中，表头使用弱强调底色与主色文本
- 引用块应以主色边框与低饱和背景进行提示，不破坏原有卡片层次
- 明暗主题切换时，Markdown 内容应自动继承全局 CSS 变量，而不是维护独立色板

### 6.2 Markdown fallback 约定

当前前端在无法使用外部 Markdown 解析器时，fallback 渲染器至少应保证以下结构可读：

- 标题
- 段落
- 引用
- 无序列表 / 有序列表
- 分隔线
- 行内代码
- 围栏代码块
- 表格
- 粗体 / 斜体 / 删除线

对于超出 fallback 范围的复杂扩展语法，可退化为普通文本或较弱语义展示，但不得输出未清洗的危险 HTML。

## 7. 后续扩展建议

以下能力不是一期必需，但建议后续保留兼容空间：

### 7.1 可选字段扩展

未来远端平台可考虑增加以下字段：

| 字段名 | 类型 | 说明 |
| :-- | :-- | :-- |
| `updated_at` | `string` | 通知更新时间 |
| `summary` | `string` | 摘要文案 |
| `link` | `string` | 外部详情链接 |
| `severity` | `string` | 明确的严重等级 |
| `pinned` | `boolean` | 是否置顶 |
| `version_range` | `string` | 版本适用范围 |

### 7.2 接口能力扩展

未来远端平台可考虑增加：

- 按时间增量拉取
- 按版本过滤
- 分页 / 游标
- 已读状态服务端同步

---

## 8. 兼容性声明

本规范以“当前远端接口返回 JSON 数组”作为一期实现基础。

只要远端继续满足以下最低条件，插件即可稳定接入：

- 返回数组结构
- 每项具备 `id`、`title`、`content`、`type`、`created_at`、`is_active`（`content_format`、`updated_at` 为可选）
- 时间字段为合法 ISO 8601 字符串

---

## 9. 团队协作约定摘要

为了确保后续团队协作一致，建议统一遵守以下原则：

1. 外部平台统一使用标准 `type` 代码
2. 插件后端统一过滤 `is_active === true`
3. 插件前端不直接展示原始 `type` 字符串
4. 已读状态默认由插件本地管理
5. 优先通过 `content_format` 显式声明正文格式（`text` / `markdown`）
6. 当前仅保留最小必要别名兼容；若后续确有新增历史值，再按需扩展别名表
7. 若新增新的 Markdown 展示元素或视觉映射规则，应同步更新前端样式与本规范
8. 新增类型时，应先更新本规范，再同步更新前端映射

---

## 10. 当前版本标准结论

当前通知系统标准可以总结为：

- 接口：无鉴权 `GET` JSON 数组，请求携带 `plugin_version`
- 数据源：外部通知平台
- 过滤：仅展示 `is_active === true`
- 排序：按 `created_at` 倒序
- 内容：支持 `text` 与 `markdown`，并兼容明显 Markdown 块级语法自动识别
- 渲染：优先使用安全 Markdown 解析链路，异常时回退 fallback 渲染器
- 主题：Markdown 内容复用管理端现有明暗主题变量，不单独维护割裂色板
- 映射：统一由前端按标准类型表展示
- 已读：插件本地维护
- 广播：由插件后端通过 WebSocket 推送前端更新
